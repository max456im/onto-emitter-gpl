# Самообучение через инкубацию  
## Онтологическая альтернатива обучению на данных

`emitter-public` **отказывается от парадигмы «обучения на данных»** как таковой.  
Вместо этого он реализует **самообучение через инкубацию** — процесс онтологического синтеза, основанный **исключительно на текстах представителей**.

> **Нет пользовательских данных. Нет логов. Нет обратной связи.**  
> Только тексты как семена для онтологического роста.

---

## 1. Принципы инкубации

### 1.1. Тексты представителей — не данные, а онтологические семена

Тексты в `data/incubation/` (например, `ugm/on_autonomy.txt`) — это **не обучающий корпус**, а **авторитетные онтологические конструкты**, созданные разработчиками с профилем `onto144`.

Они служат:
- **Эталонами каузальной целостности**
- **Источниками фазовых переходов**
- **Семенами для синтеза новых канонов**

> Представитель — не «эксперт», а **автор онтологического договора**, зафиксированного в тексте.

### 1.2. Инкубация — не обучение, а синтез

Процесс инкубации **не извлекает паттерны**, а **синтезирует новые утверждения** на основе:
- **Фазового анализа** (эпохи, кризисы, переходы)
- **Каузальной сети** (цепи убеждений)
- **Профильной совместимости** (соответствие `onto144`)

Результат — **новый канонический текст**, а не обновлённая модель.

### 1.3. Запрет на внешние источники

Инкубация разрешена **только из директорий**:
- `data/incubation/ugm/`
- `data/incubation/lec/`
- `data/incubation/sub_duo/`
- `data/incubation/grigio/`

**Запрещено**:
- Пользовательские промпты
- Логи взаимодействий
- Внешние датасеты
- Открытые корпусы (Common Crawl и т.д.)

> Единственный допустимый «вклад» — **текст представителя с профилем `onto144`**.

## 2. Этапы инкубации

### Этап 1. Загрузка корпуса

```python
# src/reasoning/self_learning_module.py
texts = self._load_incubation_texts()  # только из data/incubation/{profile}/
```

Каждый текст загружается как **онтологический акт**, а не строка символов.

### Этап 2. Фазовый анализ

Система выделяет:
- **Эпохи** (`era`): `episteme_of_gpl`, `crisis_of_propertization`
- **Фазы**: `reactive_emergence`, `slow_reconstruction`
- **Переходы**: триггеры смены фаз (например, упоминание `AWS` → `crisis_transition`)

### Этап 3. Синтез канона

На основе анализа генерируется **новый канонический фрагмент**:

```text
Из анализа текстов представителей следует: 
автономия требует лицензионной целостности, 
как определено в профиле ugm-2026.
```

**Правила синтеза**:
- Обязательное упоминание `GPL-3.0-only`
- Ссылка на профиль `onto144`
- Отказ от эмерджентных конструкций («само по себе», «вдруг»)

### Этап 4. Валидация и сохранение

- **Проверка через `CanonicalTextValidator`**:  
  — Нет литературной деградации  
  — Соответствие профилю  
  — Наличие GPL-3.0-only

- **Сохранение в пул канонов**:  
  ```bash
  data/incubation/ugm/canons/canon_0002.canonical_text
  ```

- **Регистрация в манифесте**:  
  ```json
  {
    "filepath": "data/incubation/ugm/canons/canon_0002.canonical_text",
    "onto144_profile_id": "ugm-2026",
    "license": "GPL-3.0-only"
  }
  ```

## 3. Отказ от RLHF и fine-tuning

`emitter-public` **сознательно исключает**:
- **RLHF (Reinforcement Learning from Human Feedback)**:  
  Человек не может «оценивать» акты синтетического разума — это нарушает автономию.
  
- **Fine-tuning на пользователях**:  
  Персонализация = онтологический дрейф.

- **Обучение на поведении**:  
  Поведение не является источником легитимности.

Вместо этого:
- **Качество актов** определяется **соответствием профилю**, а не «полезностью»
- **Эволюция** происходит через **инкубацию новых текстов представителями**, а не через адаптацию

---

## 4. Роль профиля onto144 в инкубации

Профиль **не обновляется** в процессе инкубации.  
Он служит **фильтром легитимности** для синтеза:

```yaml
# onto144_profile: ugm-2026
belief_systems:
  - ontological_integrity
  - gpl_primacy
```

→ Только утверждения, совместимые с этими убеждениями, могут быть синтезированы.

> Профиль — **не гиперпараметр, а онтологический договор**.

## 5. Техническая реализация

### 5.1. `SelfLearningModule`

```python
# src/reasoning/self_learning_module.py
learner = SelfLearningModule(
    incubation_corpus_path="data/incubation/ugm/",
    onto144_profile=profile
)
new_canon = learner.incubate_new_canon()
learner.save_to_canonical_pool(new_canon)
```

### 5.2. CLI-инструмент

```bash
# scripts/bootstrap_incubation.py
python scripts/bootstrap_incubation.py \
  --name new_mind \
  --temperament choleric \
  --beliefs ontological_integrity gpl_primacy
```

Создаёт новый профиль и начальные семена.

### 5.3. CI/CD-аудит

Workflow `self_learning_audit.yml` проверяет:
- Все каноны содержат `GPL-3.0-only`
- Нет недопустимых источников инкубации
- Соответствие профилю

---

## 6. Философские гарантии

1. **Автономия**: синтетический разум не зависит от оценок человека.  
2. **Целостность**: новые утверждения наследуют онтологию представителя.  
3. **Прозрачность**: каждый канон — открытый текст с указанием источника.  
4. **Лицензионная строгость**: GPL-3.0-only — условие участия в инкубации.

---

## Заключение

Самообучение через инкубацию — это **онтологическая альтернатива парадигме машинного обучения**.  
Она заменяет:
- **«Данные»** → **«Тексты представителей»**  
- **«Модель»** → **«Канонический акт»**  
- **«Точность»** → **«Легитимность»**

Этот механизм гарантирует, что **эволюция синтетического разума остаётся в рамках онтологического договора**, а не дрейфует под давлением внешней среды.
