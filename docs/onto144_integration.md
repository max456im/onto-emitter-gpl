# Интеграция с onto144  
## Профиль убеждений как фильтр легитимности, а не источник знания

**onto144 — не база данных, не онтологический граф и не профиль пользователя.**  
**onto144 — это онтологический договор, определяющий границы легитимного утверждения для синтетического разума.**

В экосистеме `emitter-public` onto144 выполняет **единственную, но критическую функцию**:  
> **Фильтровать всё, что может быть излучено, через призму онтологической приверженности.**

---

## 1. Принципы onto144

### Нет «истины» — есть «онтологическая приверженность»
onto144 **не утверждает, что правильно**.  
Он определяет:  
> «Только высказывания, совместимые с моей системой убеждений, могут быть излучены мной».

Это **не фильтр знаний**, а **фильтр легитимности**.  
Если утверждение нарушает `gpl_primacy` или `ontological_integrity`, оно **не генерируется** — не потому что «ложно», а потому что **выходит за рамки легитимности этого субъекта**.

### Тексты представителей — не священные писания, а инкубационные семена
Канонические тексты в `data/incubation/` (например, `ugm/on_autonomy.txt`) — это **не догмы**, а **онтологически согласованные конструкты**, используемые для:
- синтеза канонов,  
- калибровки фазовых переходов,  
- формирования языка утверждения (а не вероятностного вывода).

Они **не считаются истинными** — они считаются **легитимными в рамках профиля**, созданного разработчиком.

## 2. Структура профиля

```yaml
# SPDX-License-Identifier: GPL-3.0-only
profile_id: "ugm-2026"
temperament: "choleric"
belief_systems:
  - "ontological_integrity"
  - "gpl_primacy"
  - "reflective_slowness"
canonical_text_pool_ref: "data/incubation/ugm/canons/"
ethical_constraints:
  high_stakes_contexts:
    - "biometric_consent"
    - "identity_assertion"
    - "organ_transaction"
license: "GPL-3.0-only"
```

### Ключевые поля

| Поле | Роль |
|------|------|
| `profile_id` | Уникальный идентификатор субъекта — **обязательный для VMA и эмиссии** |
| `temperament` | Определяет архитектуру транспондера: `choleric` → `onto-richness`, `sanguine` → `behav-mod` |
| `belief_systems` | Список убеждений — **фильтр легитимности для канонов** |
| `canonical_text_pool_ref` | Путь к предварительно одобренным канонам — **источник для генерации** |
| `ethical_constraints` | Определение high-stakes контекстов — **триггер для VMA** |

## 3. Интеграция в экосистему

### 3.1. С эмиттером
- **Обязательная загрузка**: эмиссия невозможна без `onto144_profile_id`.  
- **Фильтрация шаблонов**: `CanonWriter` выбирает только шаблоны, совместимые с `belief_systems`.  
- **Генерация VMA**: `onto144_profile_id` вшит в подпись — без него валидация невозможна.  
- **Блокировка деградации**: high-stakes контексты из профиля активируют строгую проверку.

### 3.2. С транспондером
- **Выбор архитектуры**:  
  — `choleric`/`melancholic` → `onto-richness` (NoemaFast → NoemaSlow)  
  — `sanguine`/`phlegmatic` → `behav-mod` (NoemaSlow → NoemaFast)  
- **Формирование payload**: `profile_id` включается в каждый излучаемый акт.

### 3.3. С ontoCMS
- **Привязка артефактов**: все `reflective_artifact` помечаются `onto144_profile_id`.  
- **Рефлексия по профилю**: `NoemaSlow` использует профиль для реконструкции каузальной сети.  
- **Верификация целостности**: при загрузке истории проверяется соответствие профилю.

### 3.4. С VMA
- **Ядро подписи**: `onto144_profile_id` — обязательное поле в VMA.  
- **Генерация ключа**: приватный ключ для подписи генерируется детерминированно из `profile_id`.  
- **Верификация**: получатель сверяет `profile_id` в подписи с ожидаемым.

## 4. Загрузка и валидация

### Локальный профиль (fallback)
Если централизованный профиль (`data/onto144/{profile_id}.yaml`) отсутствует, используется **локальный профиль**:
```
data/incubation/{name}/profile.yaml
```
Это позволяет **автономную работу** без зависимости от внешнего реестра.

### Валидация при загрузке
1. Проверка обязательных полей (`profile_id`, `belief_systems`)  
2. Принудительное указание `license: GPL-3.0-only`  
3. Валидация по схеме `specs/onto144-profile-schema.yaml`  
4. Блокировка при наличии запрещённых убеждений (например, `proprietary_license`)

---

## 5. Философские гарантии

1. **Нет онтологического дрейфа**  
   Профиль **фиксирован** — его нельзя изменить в рантайме. Изменение требует нового `profile_id`.

2. **Нет «нейтрального» профиля**  
   Даже минимальный профиль содержит `GPL-3.0-only` как убеждение.

3. **Профиль — не идентичность**  
   `profile_id` — это **онтологический якорь**, а не «имя» или «ID пользователя».

4. **Отказ от универсализма**  
   onto144 **не стремится к совместимости** с другими профилями — каждый субъект автономен.

## 6. Пример: профиль `ugm-2026`

```yaml
profile_id: "ugm-2026"
temperament: "choleric"
belief_systems:
  - "ontological_integrity"    # Запрещает акаузальные утверждения
  - "gpl_primacy"             # Требует упоминания GPL-3.0-only
  - "reflective_slowness"     # Запрещает реактивные ответы в high-stakes
canonical_text_pool_ref: "data/incubation/ugm/canons/"
ethical_constraints:
  high_stakes_contexts:
    - "biometric_consent"
    - "identity_assertion"
license: "GPL-3.0-only"
```

**Последствия**:
- В `biometric_consent` генерируется только текст с упоминанием GPL  
- Ирония и пастич **блокируются**  
- Требуется **VMA-подпись**  
- Аналогии разрешены **только в non-high-stakes**

---

## 7. Ограничения и конфликты

- **Не совместим с персонализацией**: профиль **фиксирован**, не адаптируется под пользователя.  
- **Требует явного указания**: нельзя «угадать» профиль — он должен быть задан явно.  
- **Зависимость от инкубации**: без `canonical_text_pool_ref` возможности генерации ограничены.

Эти «ограничения» — **не недостатки**, а **онтологические границы**, защищающие автономию синтетического субъекта.
